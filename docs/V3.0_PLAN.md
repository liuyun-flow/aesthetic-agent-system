# V3.0 实施计划 — 闭环（Curriculum + Grounding）

> 状态：规划中，**已顺延**。原为 V2.5，因优先「信心（质量与可靠性）」版而把此闭环功能集推迟到 ~V3.0；信心版现占用 V2.5 版号（见 [V2.5_PLAN.md](V2.5_PLAN.md)）。
> 主轴：把「工具」变成「训练计划」。本计划只覆盖闭环功能集。
> 前置：V2.4 信任度量已完成（评测台在 `backend/evals/`）；建议在 V2.5 信心版的 CI/评测护栏就位后再做（grounding 改动评分，护栏能自动防回退）。

---

## 1. 目标

V2.4 让度量可信，本版让训练**成体系、有依据**：

1. **首次运行预置案例库** —— 解决冷启动：新装用户案例库为空，最有价值的「参考对比」恰恰在第一印象形成时最弱。
2. **top-N 案例注入 analyze/critique grounding** —— V2.2.1 起规划的 Phase 2：判断锚定在具体范例上，而非只有抽象规则。
3. **结构化课程** —— 用多周引导式计划替代静态主题列表，并把评估推荐接到「下一个练习」。

---

## 2. 当前状态（精确）

- **案例库**：`ReferenceCase`（title/category/aesthetic_level/style_tags/target_audience/price_band/image_id/image_description/ai_description/premium_sources/cheapness_sources/learn_from_this/avoid_copying/score）。CRUD + 语义搜索（需 OpenAI key）+ `find_cases_for_comparison`（无 key，按 category/level 取均衡组合）。
- **质量门槛**：`case_quality.is_training_ready` 要求 **completeness≥75 + image_id + level + 描述 + (premium 或 learn)**。注意：**强制要求 image_id**。
- **grounding 注入点**：`AnalyzerAgent.run(work_description, image_description=None)` / `CriticAgent.run(..., temperature=None)`——系统提示词已含 `design_knowledge`，但**都没有接收参考案例的入口**。要 grounding 必须给 run() 加参数，并由 analyze/critique 端点检索后传入。
- **训练主题**：`TRAINING_THEMES`（6 个静态字符串）+ `TodayTrainingResponse{theme, tasks}` + `/training/today`。无计划/进度/排期概念。
- **评估推荐**：`compute_report` 产出 `recommended_themes` / `next_training_plan`（纯文本，未连向任何练习入口）。

## 3. 关键设计决策（含推荐默认值）

| # | 决策 | 选项 | 推荐 |
|---|------|------|------|
| D1 | 种子案例来源（版权） | (a) 随包附图（版权风险）；(b) 纯文字标注种子（无图） | **(b) 纯文字种子**，规避版权；图片由用户后续自配 |
| D2 | grounding 可用门槛 | (a) 复用 is_training_ready（**要 image**）；(b) 新增 is_grounding_ready（不要 image，只要 level+描述+标注） | **(b)**。grounding 用的是文字标注不是像素，纯文字种子才能立即可用、解决冷启动 |
| D3 | grounding 检索方式 | (a) 语义搜索（需 key）；(b) find_cases_for_comparison（无 key） | **配置了 key→语义；否则→category/level 回退**。只取 grounding-ready 案例 |
| D4 | 注入几条 / 注入什么 | N=2–4；注入 title+level+premium/cheapness/learn（**不注入图**） | **N=3**，紧凑文本，控 token；analyze 与 critique 都注入 |
| D5 | grounding 默认开关 | (a) 有可用案例就自动开；(b) 配置 flag | **flag `GROUNDING_ENABLED`，默认开**；评测台需能 with/without 对比，故必须可关 |
| D6 | 种子可区分/可删 | (a) 加 `origin` 列（seed/user）；(b) 不区分 | **(a) origin 列**：种子可识别、可一键移除、导出时可选择排除 |
| D7 | 课程形态 | (a) 固定 prescriptive；(b) 自适应 | **prescriptive 起步 + 轻自适应**（评估推荐决定推荐哪个计划/主题），完整自适应留待后续 |
| D8 | 种子导出行为 | (a) 导出含种子；(b) 默认排除种子 | **(b) 默认排除 origin=seed**，避免把内置种子当用户数据来回搬 |

## 4. 工作流 A —— 预置案例库（冷启动）

### 4.1 种子数据
- `backend/app/db/seed_cases.json`：~9–12 条**纯文字**标注案例，覆盖 high/medium/low × 几个品类（海报/电商主图/品牌/UI），字段齐备（level + image_description 文字 + premium/cheapness/learn_from_this + target_audience + price_band）。
- 内容为合成/原创描述（规避版权），与 evals 金标准同源风格，质量过 grounding-ready 门槛。

### 4.2 幂等加载
- 启动时（init_db 之后）：若 `reference_cases` 为空**且** config `setup.seeded` 未置位 → 批量插入种子，置 `setup.seeded=done`。
- 只播一次；用户删光也不重播（靠 seeded 标记）。提供「恢复内置种子」入口（可选，后端端点）。

### 4.3 schema
- `ReferenceCase` +`origin` 列（`"user"` 默认 / `"seed"`），`_migrate_v3_0` 加列。
- `is_grounding_ready(case)`（D2）：level 有意义 + 有描述 + (premium 或 cheapness 或 learn)，**不要求 image_id**。

---

## 5. 工作流 B —— grounding 注入

### 5.1 检索
- 新 `reference_service.find_grounding_cases(db, work_description, image_description, *, n=3)`：
  - 配置了 embedding key → 语义搜索（query=作品描述）取 top-N grounding-ready；
  - 否则 → `find_cases_for_comparison` 变体，仅取 grounding-ready，均衡 high/低对比。

### 5.2 注入
- `AnalyzerAgent.run` / `CriticAgent.run` 新增 `reference_cases: list[dict] | None`。
- 在 user prompt 末尾加一段紧凑「参考范例」块（每条：等级 + 标题 + 高级/廉价感来源 + 值得学习），并指示：**以这些范例校准你的判断与评分，但评的是当前作品**。不注入图片。
- analyze/critique 端点：`GROUNDING_ENABLED` 开 → 检索并传入；关或无可用案例 → 现状行为不变。

### 5.3 与评测台的强耦合（必须遵守）
- grounding **会改变评分** → 启用前必须先有 V2.4 基线（复审 M-1）。
- 评测台加 `--grounding on|off`（或环境开关）以**对比 with/without**；用数据决定 N、是否默认开。
- 这是排序铁律：**先基线 → 再开 grounding → 再对比**。V2.5 信心版的 nightly/release 评测护栏能让这一步自动化。

## 6. 工作流 C —— 结构化课程

### 6.1 数据模型（新表，`_migrate_v3_0`）
- `training_programs`（内置定义，可种子化）：id/title/description/theme_sequence(JSON)/total_days。
- `program_progress`：id/program_id/current_step/started_at/completed_steps(JSON)/status。单用户，单活跃计划即可。

### 6.2 端点
- `GET /training/programs`（列出内置计划）· `POST /training/programs/{id}/enroll` · `GET /training/program/active`（当前计划 + 当前步骤的主题/任务）· `POST /training/program/advance`（完成当前步进入下一步）。
- `/training/today` 改造：有活跃计划→返回计划当前步；否则→回退现有静态主题轮换。

### 6.3 评估 → 练习闭环
- `compute_report` 的 `recommended_themes` 增加结构化映射：每条推荐带可点击的「去训练该主题」目标（theme key 或 program id）。
- 前端评估页推荐项 → 按钮跳工作台并预置主题（复用 V2.2 prefill 思路）。

## 10. 内部里程碑（建议顺序）

1. **N1 种子 + grounding-ready**（工作流 A）：seed_cases.json + 幂等加载 + origin 列 + is_grounding_ready + 测试。
2. **N2 grounding 注入**（工作流 B）：检索 + run() 注入 + flag + 端点接线 + 回归测试；评测台加 with/without 开关。
3. **（关口）用真实 key 跑评测台**：先 M-1 基线 → 开 grounding 对比，用数据定 N/默认值。
4. **N3 课程**（工作流 C）：数据模型 + 端点 + /training/today 改造 + 评估推荐结构化。
5. **N4 前端**：计划 UI + grounding 标注 + 种子徽标 + 评估推荐可点击 + i18n。
6. **N5 收尾**：版本 v3.0.0、迁移、文档同步、RELEASE_CHECKLIST。

> N1 先行：grounding（N2）要有案例可锚定，种子同时解决冷启动与 grounding 数据源。N3 课程相对独立，可并行或后置。
