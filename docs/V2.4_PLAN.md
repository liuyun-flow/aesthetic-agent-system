# V2.4 实施计划 — 信任度量（Evaluation Integrity）

> 状态：**已完成（V2.4.0，220 tests passed）**。下方计划保留作为实现记录。
> 实际交付 vs 计划：M1–M3、M5、M6 全部完成；M4（评测台实跑基线）需真实 key，命令已就绪由用户运行。
> 主轴：**先让度量可信，再做大**。本计划只覆盖 V2.4，不含 V2.5+。

---

## 1. 目标

让"系统能测量你的审美判断力进步"这个核心承诺**真正成立**。今天它只是看起来成立。

具体三件事：
1. 用 **LLM 评判 + 锚定评分标准** 替代 `assessment.py` 里的关键词频率维度评分。
2. 建一个 **校准/评测台**：用金标准集衡量评分器是否可信，钉死 prompt 版本以发现回归。
3. **Vision 直评**：有 vision key 时让评分走多模态，绕过"图→文"信息瓶颈。

---

## 2. 当前的问题（精确陈述）

**问题 A — 维度评分是噪声。** `compute_dimension_scores`（[assessment.py:366](../backend/app/services/assessment.py)）对记录文本数关键词命中数，命中越多分越低（30–100 线性）。这测的是"某个词出现的频率"，不是审美判断力。

**问题 B — 真实数据被丢弃。** critic 实际产出 `dimensions`（color/composition/typography/material/emotion/brand_sense，1–10）存在 `result_json` 里，但评估层完全不用，转而去数关键词。

**问题 C — 两件事被混为一谈。** 系统从没区分：
- **作品质量分**（AI 给设计打分）——critic 已产出。
- **用户判断准确度**（用户的判断与 AI 的差距）——这才是"能力"度量，今天只有一个整体 `score_gap`，没有分维度。

**问题 D — AI 分覆盖不全。** `ai_score` 仅在「有自评 + critique」时写入；analyze/iterate、无自评的记录没有任何 AI 分。维度聚合和校准都需要更稳定的 AI 分来源。

**问题 E — 评分器自身从未被校准。** "判断差距"把 AI 分当真理，但 AI 分准不准从没验证过。若 critic 系统性偏差，整条训练信号都是歪的。

## 4. 工作流 A — LLM 评判的维度评估

### 4.1 明确"测什么"
V2.4 把两个量分开、都持久化到每条记录：
- **AI 作品维度分**（`ai_dimension_scores`）：critic 已产出的 6 维 1–10，归一化存储。
- **用户判断准确度**（`judgment_accuracy`）：用户分与 AI 分的接近度。整体已有（score_gap）；V2.4 增加可选的分维度差距（需用户分维度输入，见 §4.4 渐进方案）。

### 4.2 数据存储（新增列，启动时自动迁移 `_migrate_v2_4`）
`TrainingRecord` 增加：
- `ai_dimension_scores` (JSON, nullable)：`{"color": 7, "composition": 5, ...}`，归一化到 0–100 或保留 1–10（统一一种，建议 0–100）。
- `ai_overall_score` (Integer, nullable)：稳定的 AI 总分来源，analyze/critique 都填（critique 用 total_score；analyze 用一个轻量评分调用或从维度均值推导——见 D 决策细化）。
- `eval_prompt_version` (String, nullable)：记录这条评分用的 prompt 版本，便于回溯。

> 迁移遵循现有 `_migrate_v2_3` 的加列模式（additive、幂等、旧数据安全）。

### 4.3 评分来源
- critique：直接用 critic 输出的 `dimensions` + `total_score`，归一化后存。**不额外调用 LLM。**
- analyze：当前不打分。方案：要么复用 critic 的维度评分逻辑做一次轻量评分，要么不存维度分（保持 analyze 纯定性分析）。**建议：analyze 不强制打分**，维度度量主要来自 critique 记录，避免给 analyze 加成本与语义错位。
- 评分一律走已注入 `design_knowledge` 的 SCORING_RUBRIC（V2.2.1 已就位）。

### 4.4 改造 `assessment.py`
- `compute_dimension_scores`：优先**聚合存储的 `ai_dimension_scores`**（按维度求均值、算趋势=近 7 天 vs 整体）。当没有任何带维度分的记录时，**降级到现有关键词法**（保留函数，不删）。
- `compute_overview` / `compute_mistake_patterns`：本期可不动（gap 统计是定性的、可信）；mistake_patterns 的关键词法标注为"启发式"，留待 V2.5+ 升级。
- 维度 key 对齐：现有 7 维（assessment）与 critic 6 维（color/composition/typography/material/emotion/brand_sense）不完全一致——需要一张映射表，或把 assessment 维度收敛到 critic 维度 + 商业维度。**这是一个要敲定的小设计点。**

### 4.5 接口
`/assessment/dimensions` 响应 schema（`DimensionAssessment`）**不变**——纯内部实现替换，前端 V2.2 的雷达图/进度条无需改动。

## 5. 工作流 B — 校准 / 评测台

目标：回答"critic 的分可信吗？"并在每次改 prompt / 知识库后发现回归。

### 5.1 目录与产物（新增，不进运行时镜像）
```
backend/evals/
  gold/                 # 金标准集
    pairs.jsonl         # 成对/排序标注：{"a": <work_a>, "b": <work_b>, "better": "a", "dim": "overall", "note": "..."}
    items.jsonl         # 单作品 + 粗档位标签（high/medium/low），用于分档校准
  run_eval.py           # 跑评测：对每个 item/pair 调 critic，算指标
  reports/              # 输出报告（git 忽略或保留最近一份）
```

### 5.2 金标准内容（**需用户参与采集**）
- 来源要避开版权问题：用**公认强/弱**的设计（获奖 vs 明显劣质）做粗档位标签；或纯文字描述的合成样本。
- 规模起步：~20–30 个 pair + ~15 个分档 item 足以发现明显回归。
- 标注形式：成对"谁更好"（D3），比绝对分可靠。

### 5.3 指标
- **排序相关性**：critic 总分对 pair 的判对率；分档 item 的 Spearman/Kendall。
- **分档准确率**：critic 把 high/medium/low 分对的比例。
- **分维度一致性**（如果 gold 标了维度）：每维的判对率。
- **可选**：LLM-as-judge 元评测（critic 的 rationale 是否引用了具体元素、是否符合 rubric）——非确定性、付费。

### 5.4 Prompt 版本钉死
- 在 `design_knowledge.py` 增加 `PROMPT_VERSION = "v2.4.0"` 常量；critic/analyzer 在输出/日志里带上它，评分时写入 `eval_prompt_version`。
- 评测报告头部记录 PROMPT_VERSION + 模型名 + 日期，便于版本间对比。

### 5.5 运行方式
- 纯手动 / nightly，需真实 DeepSeek（及可选多模态）key；**不 gate PR**。
- 设阈值（例：pair 判对率 ≥ 0.75）作为"发布前建议核对"项，写进 RELEASE_CHECKLIST。

## 6. 工作流 C — Vision 直评（待 D4 拍板）

现状：`describe_image` 把图变成文字 → DeepSeek 文本推理。评分器看不到像素。

两条路：
- **(a) 评分改走多模态**：当记录有 image 且配置了多模态 key 时，critic 走 GPT-4o/Claude，直接看图打分。优点：信息无损、质量上限最高。代价：破坏 DeepSeek=推理的清晰分层、更贵、加 provider 耦合、critic 需要新的图像输入路径与模型分支。
- **(b) 强化文字桥**：仍 DeepSeek，但喂更结构化的视觉负载（更细的 VisionDescription）。优点：保持分层、便宜。代价：信息瓶颈仍在，是"改良"不是"突破"。

**推荐**：本期做 (a) 但**设为可选开关**（`SCORING_VISION_DIRECT` + 多模态 key 探测），无 key 时回落到现有文本路径。这样既拿到突破、又不破坏无 key 可跑。**请确认是否接受新增多模态 provider 依赖。**

## 7. 测试策略

- **保持 216 现有测试绿**：新 LLM 调用一律 mock（沿用 test_api.py 的 Mock\*Agent 模式）。
- `test_assessment.py`（~20 用例，断言关键词法的确定性输出）：维度评分改为聚合后，这些断言需要更新为"有存储维度分时聚合、无则降级"。**降级路径必须保留对应测试**，证明无 key 行为不变。
- 新增确定性测试：给定若干带 `ai_dimension_scores` 的记录，断言聚合/趋势正确（不调 LLM）。
- 评测台（backend/eval）独立，不进 pytest 套件（付费 + 非确定性）。
- 迁移测试：旧 DB（无新列）启动后自动加列、旧数据可读。

## 9. 内部里程碑（建议顺序——评测台优先）

1. **M1 评测台骨架**（工作流 B）：金标准 schema + run_eval.py + 指标 + PROMPT_VERSION 钉死。先有"尺子"。
2. **M2 持久化 AI 维度分**（工作流 A §4.2–4.3）：加列 + 迁移 + critique 存维度分（不调新 LLM）。
3. **M3 聚合替换**（工作流 A §4.4–4.5）：assessment 聚合存储分 + 关键词降级；更新/新增测试。
4. **M4 用 M1 验证 M2/M3**：跑评测台，确认 critic 分在金标准上达阈值；记录基线报告。
5. **M5 Vision 直评**（工作流 C，待 D4）：多模态开关 + 回落；再跑一次评测台对比 (a) vs 文本路径，用数据决定是否默认开启。
6. **M6 收尾**：版本 → v2.4.0、文档同步、RELEASE_CHECKLIST 增评测项。

> M1 先行的理由：grounding（V2.5）和后续所有 prompt 改动都靠这把尺子防回归；先有尺子，M2–M5 才能用数据而非感觉来判断好坏。

## 10. 决策已定（2026-06-14）

1. **D4 / Vision 直评**：做，但**作为可选功能**——`SCORING_VISION_DIRECT` flag + 多模态 key 探测，无 key 回落现有文本路径。
2. **金标准来源**：**我先用合成样本搭骨架，用户后续替换**真实"公认强/弱"样本（纯文字描述以规避版权）。
3. **维度对齐**：收敛到 **critic 6 维（color/composition/typography/material/emotion/brand_sense）+ 2 个商业维度（价格感 price_perception / 商业适配 commercial_fit）= 8 维。**
4. **analyze 是否打分**：**否。** 维度度量只来自 critique 记录，analyze 保持纯定性分析。
