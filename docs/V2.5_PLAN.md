## 3. 关键设计决策（已定 2026-06-15）

| # | 决策 | 结论 | 理由 |
|---|------|------|------|
| D1 | CI 平台 | **GitHub Actions** | 仓库在 GitHub，原生免费 |
| D2 | 前端测试栈 | **Vitest + React Testing Library + Playwright** | 现代默认，Node 已就绪 |
| D3 | 付费评测节奏 | **仅 on-release（发布前），不 nightly、不 gate PR** | 成本最低（约每发布几分钱~$2）；mocked 测试 gate PR，付费评测只在发布前跑 |
| D4 | 缓存范围 | **只缓存视觉描述，不缓存评分** | 评分方差是信号、且缓存会破坏「再练一次」对比 |
| D5 | 缓存存储 | **SQLite 表** | 单用户本地，随库走、可清 |
| D6 | 遥测落点 | **DB 表 + 设置页汇总** | 持久化 + 累计 token/调用/平均延迟可见；每次结果脚注可选 |
| D7 | Node 版本对齐 | **CI 用 Node 20 矩阵对齐 Docker**；本机 24 仅开发便利 | Next 14 两者皆可 |

## 4. 工作流

### A. CI 脚手架（其它项的前置）
- GitHub Actions：PR 触发 → 后端 `pytest app/tests/`（mocked，无需 key）+ 前端 `npm ci && npm run build`（+ lint）。
- Node 20 对齐 Docker；Python 3.11。
- **不**在 PR 上跑付费/真实 LLM。

### B. 前端测试 + E2E
- 组件测试（Vitest + RTL）：覆盖关键交互——历史详情弹窗、判断差异、prefill「再练一次」、上传（粘贴/拖拽）、折叠面板。
- 一条 Playwright E2E 串起关键流程。
- *关键问题*：E2E 需**启动整套栈**（后端 + 前端 + 预置 DB），需要一个编排脚本/compose profile。

### C. 评测进 CI（productionize V2.4 harness）
- 扩 `backend/evals`：质量启发式（评语是否引用具体元素？分数是否在范围？是否遵循 rubric？），可选 LLM-as-judge。
- **仅 on-release** 跑（D3），key 放 CI secrets；阈值（成对判对率 ≥0.75）作为发布建议项。
- *关键问题*：付费 + 非确定性 → 永不 gate PR。

### D. 缓存层
- 缓存键 = `(image 路径/描述 + model + PROMPT_VERSION)`；`PROMPT_VERSION` 改动即自动失效（V2.4 已有）。
- 只缓存 **VisionAdapter 的描述结果**（稳定且最贵）；评分不缓存。

### E. 成本/延迟遥测
- 在 LLM 调用处采集 `completion.usage`（token）+ 墙钟延迟 → 落 DB；设置页展示累计。
- *关键问题*：触及每个 agent 调用 → 用一个薄包装；仅能对真实 usage 对象完整验证（需 key）。
